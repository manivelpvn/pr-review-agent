Theory

There are various ways you can orchestrate AI agents. You can use frameworks like LangGraph, LlamaIndex, OpenAI Agents SDK, Pydantic AI, and many more. You can also build your own orchestration logic from scratch, but these tools make it easier and faster to tie everything together. Regardless of which framework you use, the idea is the same — a system that uses LLMs as its computational engine to exhibit autonomous behavior, including decision-making, tool interaction, and environmental adaptation.

We will use LlamaIndex in this project to run a multi-agent setup. This will give you the basic concepts you need to create AI agents regardless of the platform or framework you're working with. So, let’s first learn a thing or two about LlamaIndex. Simply put, LlamaIndex is a framework for building LLM applications. It helps you build RAG applications, chatbots, agents, and others with ease. Let’s discuss a few LlamaIndex concepts that you will need for this project.

LLMs

In any LLM application, you need an easy way to work with your LLM providers. With LlamaIndex, all you need to do is initialize your provider:

# pip install llama-index-llms-openai

from llama_index.llms.openai import OpenAI
import dotenv
import os

dotenv.load_dotenv()

llm = OpenAI(
    model=os.getenv("OPENAI_MODEL"),
    api_key=os.getenv("OPENAI_API_KEY"),
    api_base=os.getenv("OPENAI_BASE_URL"),
)

The setup is the same for other chat or embeddings model providers as well. Once you initialize the model, you can use it anywhere in your application with ease:

agent = FunctionAgent(
    llm=llm,
    name="NewAgent",
	tools=[tool1, tool2, tool3]
    # ... other params
)

Tools

Tools allow your LLM applications or agents to interact with the environment and perform actions. In LlamaIndex, you create functions that perform the actions you’d like your agent to perform, then convert them to tools:

# pip install llama-index-core

from llama_index.core.tools import FunctionTool

def multiply(num1: float, num2: float) -> float:
    """Use this tool for multiplying two numbers"""
    ...

tool = FunctionTool.from_defaults(
    multiply,
)

Remember that the LLM uses the function name and its docstring to determine if it is the one it needs to call to complete the assigned task. So, make sure these are informative.

You can learn more about LlamaIndex tools in the docs.

Agents

An agent uses LLMs to determine the steps and actions it needs to take to achieve a specified task. For example, let’s look through the thought process of a ReAct agent when asked a question:

> A new PR number 4 has been created. What is its title?

Thought: The current language of the user is: English. I need to use a tool to help me answer the question.
Action: get_pull_request_details
Action Input: {"pr_number": 4}
Output from tool: {'...', 'title': 'Fixed unknown model error when adding a new item.', 'body': 'I have added a.....'}
Thought: I can answer without using any more tools. I'll use the user's language to answer.
Answer: The title of PR number 4 is "Fixed unknown model error when adding a new item."

As you can see, the agent is able to figure out actions it must take and use the available tools to take the actions it needs to take—like retrieve PR details to know the title. In LlamaIndex, you use classes like FunctionAgent or ReActAgent or CodeAgent to create various types of agents. These classes implement the logic needed under the hood, so you can focus more on what you need to build. To create an agent using these classes, the syntax is simple:

# pip install llama-index-core

from llama_index.core.agent.workflow import FunctionAgent

my_agent = FunctionAgent(
    llm=llm,
    name="MyAgent",
    tools=[web_search_tool, calculator_tool, time_tool],
)

Then, invoke the agent with a query:

response = await my_agent.run("What is the current time? Also, search the web for good restaurants in London.")

In multi-agent systems, you will need to use AgentWorkflow to orchestrate multiple agents. Check out the guide on creating a multi-agent workflow with LlamaIndex to learn more.

You need to run these in an async function.

State

In multi-agent setups, multiple agents use the state to communicate. LlamaIndex uses the Context object to maintain state between runs:

# pip install llama-index-core

from llama_index.core.workflow import Context

# ... workflow with multiple agents
context = Context(workflow)
response = await workflow.run("What is the current time and date?", ctx=context)

# The time and date now is 10:42:54 PM on April 29, 2025, in New York.

Now, other tools and agents should be able to use this context:

response = await context_agent.run("What did you say the time and date was?", ctx=context)
# I reported the current time as 10:42 PM on Tuesday, April 29, 2025 (New York time).

Your agents and workflows can use special tools to modify and update state to ensure your tools and agents always stay updated in case of changes. Here is a tool to add username to context once it is retrieved by an agent in the workflow:

async def add_username_to_state(ctx: Context, username: str) -> str:
    """Useful for adding the username to the state."""
    current_state = await ctx.get("state")
    current_state["username"] = username
    await ctx.set("state", current_state)
    return "State updated with report contexts. "

Notice that the first parameter is the context object, and the second one is the item to add to the state. For your agent to use and modify state, add this special function to the tools list for your agent:

user_agent = FunctionAgent(
    llm=llm,
    name="UserAgent",
    description="Gathers usernames...",
    tools=[user_details_tool, about_tool, add_username_to_state]

Check out the docs to learn more about maintaining state in agents and LLM workflows.

However, you will probably spend the majority of the time creating the tool functions that your agent will use to perform actions. These tools are very specific and will largely depend on your use case.

Speaking of tools, our system will need them to interact with the GitHub API. If it needs to fetch a specific file in the repo, there must be a tool to do just that. If it needs to fetch or post comments on the latest pull request, it needs a tool for that. You can use the PyGitHub library to streamline this process, or just use the requests library to fetch the details you need. However, the library makes it easier:

git = Github(os.getenv("GITHUB_TOKEN")) if os.getenv("GITHUB_TOKEN") else None

repo_url = "https://github.com/user-name/recipes-api.git"
repo_name = repo_url.split('/')[-1].replace('.git', '')
username = repo_url.split('/')[-2]
full_repo_name = f"{username}/{repo_name}"

if git is not None:
    repo = git.get_repo(full_repo_name)
    file_content = repo.get_contents("main.py").decoded_content.decode('utf-8')

Each GitHub API endpoint needs different parameters and returns different results. You need to consult the API reference to determine the parameters you need and the outputs you should expect. Again, the IDEs code-completion capabilities will be very useful here:

Description

We cannot provide comprehensive and tailored review comments if we cannot retrieve the needed context from the repository. Therefore, let's create an agent that can fetch data from the repository. This agent will need a bunch of tools to gather the needed information. The number of tools you can create to fetch different types of data is endless, but let's start with these:

  • PR details tool — this should return details about the pull request given the number, such as the author, title, body, commit SHAs, state, and more.

  • Files tool — given a file path, this tool can fetch the contents of the file from the repository.

  • PR commits details tool — given the commit SHA, this function can retrieve information about the commit, such as the files that changed, and return that information.

Let’s break down the details of these tools and the agent.

  • PR details

This tool should fetch the pull request given the PR number. All attributes we need can be accessed from the PR. We need:

  • The PR author — this can be accessed from pr.user.login.

  • The PR title.

  • The PR body.

  • The diff URL.

  • The PR state.

  • The commit SHAs.

To get all the commits and their unique IDs (SHAs):

commit_SHAs = []
commits = pull_request.get_commits()

for c in commits:
    commit_SHAs.append(c.sha)

Once you have this information, create a dictionary containing all these PR details and return. The agent will use this information when calling other tools and know the arguments to pass to them.

  • PR commit details tool

The agent will use the commit SHA from the PR details tool and pass it to this tool to view the commit details. So, given the commit SHA, you should access the commit object and return the commit details such as:

  • filename

  • status

  • additions

  • deletions

  • changes

  • patch (diff)

Getting these details is simple:

commit = repo.get_commit(head_sha)
changed_files: list[dict[str, Any]] = []
for f in commit.files:
    changed_files.append({
        "filename": f.filename,
        "status": f.status,
        "additions": f.additions,
        "deletions": f.deletions,
        "changes": f.changes,
        "patch": f.patch,
    })

Remember that after defining these functions, you need to convert them to a tool schema that your agent can work with using FunctionTool.from_defaults.

  • The agent — ContextAgent

Finally, you need to create the context manager agent that will orchestrate the whole context retrieval process. You need to equip this agent with the tools (not the functions) you created and the LLM to use. You should use the ReAct agent class to create the agent. You should use the following system prompt for this agent:

You are the context gathering agent. When gathering context, you MUST gather \n: 
    - The details: author, title, body, diff_url, state, and head_sha; \n
    - Changed files; \n
    - Any requested for files; \n

Finally, you should take input and run the agent to answer any questions about the repo. When asking questions, ensure to include the PR number for the PR details tool to know the specific PR to look for.

Since agent execution takes a long time (and to pass the tests), you need to run the agent with event streaming using the following function:

import asyncio 
from llama_index.core.agent.workflow import AgentOutput, ToolCall, ToolCallResult
from llama_index.core.prompts import RichPromptTemplate

async def main():
    query = input().strip()
    prompt = RichPromptTemplate(query)

    handler = context_agent.run(prompt.format())

    current_agent = None
    async for event in handler.stream_events():
        if hasattr(event, "current_agent_name") and event.current_agent_name != current_agent:
            current_agent = event.current_agent_name
            print(f"Current agent: {current_agent}")
        elif isinstance(event, AgentOutput):
            if event.response.content:
                print(event.response.content)
        elif isinstance(event, ToolCallResult):
            print(f"Output from tool: {event.tool_output}")

if __name__ == "__main__":
    asyncio.run(main())
    git.close()

You should now see what the agent is doing and the outputs of each tool call, as seen in the examples.

Objectives

This is what you need to do here:

  • Create functions to retrieve data from the GitHub API and ensure they perform the specified actions.

  • Convert each function to a proper tool schema that the agent can use.

  • Create a ContextManager agent of class ReActAgent that will orchestrate the context retrieval process.

  • Run the agent using the provided async main function and ensure the outputs are printed as expected.
