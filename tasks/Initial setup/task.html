Description

Code review is a central part of every software development pipeline. It involves examining someone’s code changes by one or more developers other than the author. Reviews typically happen in a pull/merge-request workflow: you open a request with your changes, reviewers check your work, they leave comments, and—once all feedback is addressed—the code is approved and merged.

Code reviews make it easy to catch bugs early, maintain consistency, share knowledge, and improve maintainability. However, since reviews require humans to check the changes and provide comments, it may take a long time before reviewers are available to check the review. To save some time and effort, we can offload some of these routine tasks to an AI agent while allowing human reviewers to do the final checks and merging (if needed).

The agent would be able to perform common code review tasks and provide feedback. These would help catch obvious issues early and provide immediate feedback before human reviewers check the more complex issues. This is a high-level view of the architecture of the system we will be building:

Before we start developing our agent, we need to set up a repository to test it out. This demo repository helps us develop and refine our agent before using it in other code bases. The provided code is for a recipe API application built with Django and the Django Rest Framework to add, view, and manage recipes. You don't need to know Django or how this code works for this project, it is just a demo repository for testing (ideally, the agent should work with any repository given its URL).

You need to create this repository in your GitHub account so that you have full control of it. Go ahead and download the source code and extract it in a different folder from the one containing this project to avoid any conflicts. Once you’ve downloaded it, feel free to follow the instructions in the README.md file to run it locally. This is, however, not necessary.

Next, you need to create a new branch in the repo. You can name it anything, but let's assume it is a new feature/recipe-ratings, branch where a developer has added a new feature to rate recipes. Once you have the new branch, replace the contents of the following files with the given content:

app/serializers.py:

from rest_framework import serializers
from .models import Recipe, RecipeRating

class RecipeSerializer(serializers.ModelSerializer):
    class Meta:
        model = Recipe
        fields = ["id", "title", "description", "time_minutes", "price"]

class RecipeRatingSerializer(serializers.ModelSerializer):
    class Meta:
        model = RecipeRating
        fields = ["id", "stars", "created_at"]
        read_only_fields = ["id", "created_at"]

    def validate_stars(self, value):
        if not (1 <= value <= 5):
            raise serializers.ValidationError("Rating must be between 1 and 5.")
        return value

app/models.py:

from django.db import models
from django.conf import settings


class Recipe(models.Model):
    title = models.CharField(max_length=200)
    description = models.TextField(blank=True)
    time_minutes = models.IntegerField()
    price = models.DecimalField(max_digits=5, decimal_places=2)

    def __str__(self):
        return self.title


class RecipeRating(models.Model):
    recipe = models.ForeignKey(
        Recipe, on_delete=models.CASCADE, related_name="ratings"
    )
    user = models.ForeignKey(
        settings.AUTH_USER_MODEL, on_delete=models.CASCADE
    )
    stars = models.PositiveSmallIntegerField(
        choices=[(i, f"{i} star{'s' if i>1 else ''}") for i in range(1, 6)]
    )
    created_at = models.DateTimeField(auto_now_add=True)

    class Meta:
        unique_together = ("recipe", "user")
        ordering = ["-created_at"]

Commit and push these changes to your repository. Then, head over to your repository on GitHub and open a new pull request for these changes. This mirrors the typical contribution flow in GitHub, where you clone the repo, create a new branch from main, add changes, commit, push, and open a new PR. We will use this PR to check how our agent responds.

Notice that after a few seconds, a new comment is added to the PR with coverage report details. Although we won't use it, an agent can use information from this comment to know the outcome of tests.

Finally, we need to authorize the agent to post review comments for the PR once generated. Thus, let's create a new fine-grained access token and scope it to the recipes-api repository to grant these permissions. Assign repository permissions to:

  • Read and write issues — this allows the agent to both read and write issues and issue comments;

  • Read and write pull requests — this allows the agent to both read and write pull requests and pull request comments.

You can create access tokens in your GitHub account settings under developer settings:

Once generated, place the key in your .env file and name the variable GITHUB_TOKEN. That's it. Now, you're ready to create agents to retrieve this context, review, draft a comment, and post it back to GitHub.

To pass this stage, provide the full URL to your repository in the main.py file:

Objectives

Here is what you need to do:

  • Download the provided code for a recipe API application and extract it into a separate folder to avoid conflicts with the project.

  • Create a new recipes-api repository in your GitHub account and publish the application there.

  • Create a new branch and give it any name you like.

  • While in this new branch, modify the serializers.py and models.py files with the specified changes, commit the changes, and create a new PR.

  • Ensure to give the new PR a meaningful title and detailed description.

  • Generate a new fine-grained GitHub access token scoped only to the recipes-api repository and set the appropriate permissions to read and write issues and PRs. Copy the access token and place it in your .env file for this project (not the recipes-api application), and name the variable GITHUB_TOKEN.

  • Provide the full URL to your repo in the main.py file.
