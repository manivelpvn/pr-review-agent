Description

Unfortunately, we’ve had to manually run our agent when we suspect a new pull request has been created. But, what if there is a better way? What if we can ensure that the agent runs automatically whenever a new pull request is created? This is where GitHub Actions comes in.

GitHub Actions allows you to run scripts, tests, builds, and even publish projects in response to certain events in your repository, such as whenever a pull request is created. Therefore, we can leverage this to run the agent automatically whenever a new PR is created. Don’t worry, you don’t have to be a DevOps Engineer to set this up, all steps will be provided.

To start using GitHub Actions, you need to create special YAML files in the .github/workflows directory. If you look closely at the PR request we created earlier for the recipe-api application, there is already a comment from GitHub Actions containing test results:

When we opened a new PR earlier, GitHub Actions performed the actions specified in the .github/workflows/ci.yml file. Here, we instructed GitHub Actions to run pytest and coverage, then post the results as a comment in the PR. Cool right? This is a common practice in modern software development. Now, let’s add another instruction in the ci.yml file to run the agent we created. With this setup, every time a new PR is created, we will also have the agent’s review comment, in addition to the test results.

Before we set up the workflow, let’s make sure that all dependencies are in place. Remember that we orchestrated the agent with LlamaIndex and also used PyGithub to interact with the GitHub API. Also, remember that the recipe-api application uses poetry as the package manager.

Load the recipe-api app in your IDE and ensure that you're in the main branch of the repo and not a feature branch because we will be committing directly to main.

To add dependencies, open a terminal in the root directory of the recipe-api application, and run the following commands:

poetry env activate
poetry add llama-index-core
poetry add llama-index-llms-openai # If you used another provider, add the appropriate LlamaIndex package for that provider.
poetry add pygithub
poetry add python-dotenv

You need to install poetry before running these commands. See the documentation on how you can install it for your OS (you can also use pip install poetry).

Once you have added these dependencies, create a new Python file named agent.py in the root directory of the recipe-api project. You should have the following directory structure now:

recipe-api/                      # Root directory of the project
    │
    ├── .github/                 # Hidden directory for GitHub-specific files
    │   └── workflows/           # Directory for GitHub Actions workflow files
    │       └── ci.yml           # Contains workflow file ci.yml
    │
    ├── app/                     # main application directory
    ├── recipe/                  # settings and other Django file
    ├── agent.py                 # code for the PR review agent
    ├── pyproject.toml           # build system and tool configurations.
    ├──  .....                   # other files
    └── README.md                # Project description file

This file should contain the code for the three agents you’ve developed so far. Ensure that this code already works locally as expected, since GitHub Actions will run it as you would in the local environment. After you’ve added the code, we just need to instruct GitHub Actions to run this Python file and push these changes to the main branch.

Let’s now add the instructions in the ci.yml file to run the agent. We only need to add one more instruction at the bottom, right after the “Publish coverage results” step, as follows:

- name: Review agent
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
    REPOSITORY: ${{ github.repository }}
    PR_NUMBER: ${{ github.event.pull_request.number }}
    OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
    OPENAI_BASE_URL: ${{ secrets.OPENAI_BASE_URL }}
  run: poetry run python agent.py $GITHUB_TOKEN $REPOSITORY $PR_NUMBER $OPENAI_API_KEY $OPENAI_BASE_URL

Look closely at the env field. In GitHub Actions, we securely manage environment variables using secrets. This is the most secure way of handling API keys, database connection strings, and others in production environments. GitHub Actions will pass the keys you set as secrets to your code when it runs.

You can set repository secrets in the settings for that repo:

We only need to create repository secrets for the model provider. For example, in the example above, you need to set only two keys:

  • OPENAI_API_KEY — your OpenAI API key.

  • OPENAI_BASE_URL — if you are not using a custom base URL that you need to keep secret, you don’t have to set this key.

The GITHUB_TOKEN key is already available for GitHub Actions workflows. This special key is reserved, so GitHub Actions will not allow you to use that name (and you don't need to). To provide the agent with write permissions for PRs, the workflow already has the following permissions set for the ci job:

permissions:
  checks: write
  pull-requests: write

Therefore, you don’t need to perform any further actions here, but you would need to if you were authoring the workflow from scratch. If you are using a different model provider, ensure to set the API keys for that provider in secrets and also modify the env and run instructions with the names you used. For instance, if you used Groq, your setup might look like:

- name: Review agent
  continue-on-error: false
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
    REPOSITORY: ${{ github.repository }}
    PR_NUMBER: ${{ github.event.pull_request.number }}
    GROQ_API_KEY: ${{ secrets.GROQ_API_KEY }}
  run: poetry run python agent.py $GITHUB_TOKEN $REPOSITORY $PR_NUMBER $GROQ_API_KEY

In addition, we are also passing the repository and PR number to the script for the PR that triggered the workflow. Now, when we run the script, we have access to these variables as we’d normally access and use them from our local .env file:

import dotenv

dotenv.load_dotenv()

llm = OpenAI(
    model="gpt-4o-mini",
    api_key=os.getenv("OPENAI_API_KEY"),
    api_base=os.getenv("OPENAI_BASE_URL"),
)
github_client = Github(os.getenv("GITHUB_TOKEN")) if os.getenv("GITHUB_TOKEN") else Github()
repository = os.getenv("REPOSITORY")
pr_number = os.getenv("PR_NUMBER")

Of course, ensure to use the names you used in your code when you set secrets for your model provider. Also, remember you cannot take input here. So, in your main function, modify the initial query to something like:

query = "Write a review for PR: " + pr_number

That’s it. That’s all the setup you need. Once everything is in place, push the changes to your main branch. To test this setup, switch to the feature branch, make some changes, commit, and push. You can also close and re=open the existing PR or write a simple comment because the workflow is set to trigger on all PR events. You can monitor the progress in the Actions tab of the repo:

Objectives

To automate the agent you’ve been building, follow these steps:

  • Open a terminal in the root directory of the recipe-api project and run the appropriate commands to add dependencies for the agent (ensure you're in the main branch).

  • Create a new agent.py file in the root directory of the project, where the code for the three agents will live.

  • The code you paste in agent.py is the solution you’ve developed so far for the three agents. Since GitHub Actions will run exactly this code, ensure it works as expected.

  • Update the agent.py file to retrieve the repository and PR number from the environment (keep in mind that GitHub Actions passes the repository as “username/repo-name”, e.g hyperskill/recipe-api). With these updates, the code won't work locally.

  • Since we can't take input, set the initial query in the main function of agent.py as f“Write a review for PR number {pr_number}. Remember, the PR number is coming from the environment.

  • Add a new step in the ci.yml file in the .github/workflows directory with instructions to run the agent.py file and pass the appropriate environment variables to it.

  • Set the API key for your model provider in secrets. If you also need to set the base URL as a repository secret, please do so.

  • Commit and push these changes to GitHub.

  • Close and re-open the PR, write a PR comment, or create an entirely new PR to trigger the workflow, and verify that a new comment was added by GitHub Actions.

  • Add the full URL to your repository in the main.py file if it doesn't exist.
